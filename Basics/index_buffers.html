1.13 Index Buffers

In this section, we'll examine the index buffer, a crucial topic to cover before discussing model loading in the next tutorial. Previously, we explored different triangle rendering topologies, such as triangle-list, the most basic choice. However, it had a drawback â€“ redundant vertices, as many vertices were shared by multiple triangles.

While both triangle-strip and triangle-fan (triangle-fan doesn't exist) are more memory-efficient, not all objects can be easily represented using these topologies. Often, manual partitioning of 3D meshes is required.

For instance, a simple cube cannot be represented by a single triangle-strip. Instead, we need three triangle-strips to enclose the cube, with the vertical walls drawn using one strip and the top and bottom rendered separately with two additional strips. For complex 3D models like a teapot, this partitioning becomes non-trivial and results in hundreds or even thousands of pieces, which is far from ideal.

An alternative solution to this problem is the index buffer. The concept is straightforward: we maintain a pool of individual vertices, regardless of their connectivity. Then, we introduce an index buffer. Instead of storing the complete vertex data with all attributes like coordinates, colors, and texture coordinates, we only store indices into the vertex pool. The index buffer defines the connectivity of these vertices.

If a vertex is shared by multiple faces, the index buffer contains redundant vertex indices, but it never duplicates the actual vertex data. Since an index is only an integer, this approach saves significant storage space. Using the index buffer is a more generic and widely adopted solution in 3D applications.

Now, let's look at the implementation of the index buffer. Notably, the shader code remains unchanged because the graphics pipeline handles the topology and vertex processing.

    const positions = new Float32Array([
        -100.0, 100.0, 0.0,
        -100.0, 100.0, 200.0,
        100.0, 100.0, 0.0,
        100.0, 100.0, 200.0,
        100.0, -100.0, 0.0,
        100.0, -100.0, 200.0,
        -100.0, -100.0, 0.0,
        -100.0, -100.0, 200.0,
    ]);

    const positionBuffer = createGPUBuffer(device, positions, GPUBufferUsage.VERTEX);

In the JavaScript code, we first define the vertex pool, which includes all 8 vertices of the cube without any duplication, as the index buffer will eliminate redundant vertices.

    const indices = new Uint16Array([0, 1, 2, 3, 4, 5, 6, 7, 0, 1]);
    const indexBuffer = createGPUBuffer(device, indices, GPUBufferUsage.INDEX);
    const indices2 = new Uint16Array([3, 1, 5, 7]);
    const index2Buffer = createGPUBuffer(device, indices2, GPUBufferUsage.INDEX);

Additionally, we introduce two index buffers. One for the vertical walls that form a single triangle-strip like before, and another for the cube's bottom (our cube has an open lid).

The index buffer uses an integer format (Uint16Array), as 16 bits suffice for representing simple geometry like a cube.

    const pipelineDesc = {
        ...
        primitive: {
            topology: 'triangle-strip',
            stripIndexFormat: 'uint16',
            frontFace: 'ccw',
            cullMode: 'none'
        }
        ...
    };

When defining the pipeline, in addition to specifying the topology as triangle-strip, we must define the stripIndexFormat as uint16. This indicates that we'll use an index buffer for drawing.

    passEncoder = commandEncoder.beginRenderPass(renderPassDesc);
    ...
    passEncoder.setVertexBuffer(0, positionBuffer);
    passEncoder.setIndexBuffer(indexBuffer, 'uint16');
    passEncoder.drawIndexed(10);
    passEncoder.setIndexBuffer(index2Buffer, 'uint16');
    passEncoder.drawIndexed(4);
    passEncoder.end();

To issue draw commands, we first set the vertex buffer (positionBuffer) as before to supply the vertex pool. Then, for each draw call, we specify the index buffer using setIndexBuffer and call drawIndexed with the number of indices to render. The GPU pipeline uses the index buffer to retrieve vertex indices and then fetches the corresponding vertices, passing them to the shader. The result is identical to previous rendering but with less data sent to the GPU.

To clarify, while we initially discussed the limitations of using triangle-strip, in this example, we still manually partition an open cube into two triangle-strips, despite an index buffer is being used. In fact, making use of an index buffer doesn't conflict with your choice of triangle topology. The point is, even if you opt for triangle-list, with an index buffer, substantial data size savings can still be achieved.
